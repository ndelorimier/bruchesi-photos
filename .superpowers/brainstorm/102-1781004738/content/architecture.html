Architecture globale — 3 approches

Comment déployer les composants sur le DS1520+

A

Full Docker Compose sur NAS ⭐ Recommandé

Tout tourne dans des containers Docker orchestrés par un seul fichier docker-compose.yml. Un seul commande pour démarrer, redémarrer, mettre à jour.

┌─────────────── NAS DS1520+ ───────────────┐
│ nginx ← HTTPS + reverse proxy │
│ ↓ │
│ app ← Node.js API + Vue PWA │
│ ↓ ↓ │
│ postgres compreface (reconnaissance) │
│ ↓ │
│ 📁 /photos (dossier NAS natif) │
└───────────────────────────────────────────┘

Pour

  • Stack familière (Node.js)
  • Portable et sauvegardable
  • Isolation propre des services
  • Mise à jour sans downtime

Contre

  • Demande Container Manager actif
  • Légèrement plus RAM (~1.5 GB total)
B

Synology Photos + App légère par-dessus

Utilise Synology Photos comme couche de stockage. L'app Node.js se branche sur l'API Synology pour gérer les albums. Moins de containers.

┌─────────────── NAS DS1520+ ───────────────┐
│ Synology Photos (albums, accès) │
│ ↓ │
│ app ← Node.js (thin layer) │
│ ↓ ↓ │
│ sqlite compreface (Docker) │
└───────────────────────────────────────────┘

Pour

  • Moins de RAM (pas de Postgres)
  • Interface DSM pour gestion manuelle

Contre

  • API Synology Photos peu documentée
  • Couplé à Synology — difficile à migrer
  • Workflow approbation plus complexe à greffer
C

NAS photos + Backend Railway (cloud léger)

Les photos restent sur le NAS. L'API Node.js, la DB et les notifications tournent sur Railway. CompreFace seul sur le NAS.

┌─── Railway ───┐ ┌─── NAS ─────────┐
│ Node.js API │ ←→ │ CompreFace │
│ PostgreSQL │ │ 📁 /photos │
│ Push notifs │ │ (tunnel ngrok?) │
└───────────────┘ └─────────────────┘

Pour

  • Notifs/email fiables (Railway toujours dispo)
  • NAS hors-ligne = app partielle mais pas morte

Contre

  • Tunnel réseau NAS ↔ Railway complexe
  • Coût Railway supplémentaire
  • Photos quittent potentiellement le camp
