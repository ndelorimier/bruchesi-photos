Design — Section 2 : Modèle de données

10 tables PostgreSQL — structurées autour des campeurs, photos et parents

Gestion du camp
semaines
 id, nom, date_debut, date_fin

campeurs
 id, prenom, nom
 semaine_id → semaines
 compreface_subject_id
 statut: aucun|partiel|confirmé

parents
 id, email, prenom, nom
 campeur_id → campeurs
 compte_actif (bool)

employes
 id, email, prenom, nom
 role: photographe|approbateur|admin
Photos & Reconnaissance
photos
 id, fichier_path, thumbnail_path
 uploaded_by → employes
 uploaded_at
 statut: pending|approved|rejected
 approuve_par → employes

photo_tags
 photo_id → photos
 campeur_id → campeurs
 confidence (0.0–1.0)
 confirme_par_humain (bool)

face_profiles
 campeur_id → campeurs
 type: parent_submit|selfie_station
 fichier_path, enregistre_at
Auth & Notifications
magic_links
 token (UUID), parent_id → parents
 expires_at, used_at

push_subscriptions
 parent_id → parents
 endpoint, p256dh, auth
 (données Web Push VAPID)

notifications
 parent_id → parents
 photo_id → photos
 type: new_photo
 sent_at
Import CSV
Format attendu :
prenom_enfant, nom_enfant, semaine,
prenom_parent, email_parent
Un enfant peut avoir 2 parents (2 lignes).
Import idempotent — re-importer ne duplique pas.
Flux photo complet :
Upload → photos.statut = pending → CompreFace détecte → photo_tags créés → approbateur valide → statut = approved → notifications envoyées aux parents des campeurs tagués
